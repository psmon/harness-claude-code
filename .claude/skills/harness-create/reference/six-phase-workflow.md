# 6-Phase 워크플로우 (revfactory 기반)

> 출처: [revfactory](https://github.com/revfactory) — 도메인 분석 기반 하네스 설계

도메인 분석에서 출발하여 검증/배포까지 이르는 6단계 구조입니다.
새 하네스 하위 요소를 생성할 때 이 워크플로우를 따릅니다.

```
1. 도메인 분석 → 2. 팀 디자인 → 3. 에이전트 정의 → 4. 스킬 설계 → 5. 워크플로우 → 6. 검증
```

---

## Phase 1: 도메인 분석 (Domain Analysis)

**목적**: 문제 영역을 먼저 이해한 후 설계에 착수

| 항목 | 내용 |
|------|------|
| **입력** | 사용자 요구사항, 기존 하네스 현황 |
| **활동** | memorizer MCP 검색, 관련 오픈소스 분석, 도메인 범위 정의 |
| **산출물** | 도메인 분석 결과 (메모리 또는 문서) |
| **대응 디렉토리** | `knowledge/` |

**수행 방법**:
1. `mcp__memorizer__search`로 관련 키워드 검색 (한글/영문 모두)
2. 기존 `knowledge/` 내 관련 지식 확인
3. 필요 시 웹 검색으로 외부 참고 자료 수집

---

## Phase 2: 팀 디자인 (Team Design)

**목적**: 에이전트 팀 구성을 설계

| 항목 | 내용 |
|------|------|
| **입력** | Phase 1 도메인 분석 결과 |
| **활동** | 필요 에이전트 식별, 역할 배분, 협업 패턴 결정 |
| **산출물** | 팀 구성 계획 |
| **대응 디렉토리** | `agents/` |

**고려사항**:
- 최소 에이전트 원칙: 꼭 필요한 에이전트만 생성
- 기존 에이전트와의 관계 정의
- 실행 모드 선택: 에이전트 팀 모드 vs 시니어 에이전트 모드

---

## Phase 3: 에이전트 정의 (Agent Definition)

**목적**: 개별 에이전트의 역할과 능력을 상세 정의

| 항목 | 내용 |
|------|------|
| **입력** | Phase 2 팀 구성 계획 |
| **활동** | 에이전트별 역할, 능력, 도구, 제약사항 정의 |
| **산출물** | 에이전트 스펙 문서 (agents/ 하위) |
| **대응 디렉토리** | `agents/` |

**에이전트 스펙 포함사항**:
- 이름과 역할 설명
- 사용 가능한 도구 (MCP 포함)
- 입/출력 정의
- 다른 에이전트와의 관계

---

## Phase 4: 스킬 설계 (Skill Design)

**목적**: 에이전트에 부여할 스킬을 도메인 지식과 매핑

| 항목 | 내용 |
|------|------|
| **입력** | Phase 3 에이전트 스펙 |
| **활동** | 스킬 분류 (Workflow/Capability/Hybrid), 메모리 매핑 |
| **산출물** | 스킬 정의 문서 (knowledge/ 하위) |
| **대응 디렉토리** | `knowledge/` |

**스킬 분류**:
- **Workflow**: 프로세스/절차 (ex: 배포 절차)
- **Capability**: 기술/도구 능력 (ex: 코드 리뷰)
- **Hybrid**: 절차 + 능력 혼합

---

## Phase 5: 워크플로우 (Workflow Setup)

**목적**: 에이전트들의 실행 흐름을 구성

| 항목 | 내용 |
|------|------|
| **입력** | Phase 3-4 에이전트/스킬 정의 |
| **활동** | 실행 순서, 분기 조건, 패턴 적용 |
| **산출물** | 워크플로우 정의 (engine/ 하위) |
| **대응 디렉토리** | `engine/` |

**아키텍처 패턴 선택**:
- **Chain**: 순차 실행 (A → B → C)
- **Verify**: 검증 루프 (실행 → 검증 → 재시도)
- **Parallel**: 병렬 실행 (A, B, C 동시)
- **Workflow**: 조건부 분기가 포함된 복합 흐름

**데이터 프로토콜**:
- SendMessage: 실시간 에이전트 간 통신
- TaskCreate/Update: 작업 추적
- 파일 기록: Workspace/Artifacts 영속화

---

## Phase 6: 검증 (Verify & Deploy)

**목적**: 생성된 요소가 프로젝트 규칙에 맞는지 확인

| 항목 | 내용 |
|------|------|
| **입력** | Phase 1-5 전체 산출물 |
| **활동** | 3계층 배치 확인, 4종 산출물 갱신, 테스트 |
| **산출물** | 최종 검증 완료, 버전 발행 |
| **대응 디렉토리** | `docs/`, `design/` |

**검증 체크리스트**:
- [ ] 새 요소가 올바른 계층 디렉토리에 배치되었는가?
- [ ] 기존 요소와의 관계가 명확한가?
- [ ] 4종 필수 산출물이 모두 갱신되었는가?
- [ ] harness.config.json 버전이 올바른가?
- [ ] 블루프린트에 시각적으로 반영되었는가?

## Core Components

### 오케스트레이터 (Orchestrator)
에이전트들을 조율하는 중앙 컨트롤러
- SharedMemory를 통한 에이전트 간 상태 공유
- ContextWindow 관리

### Two Execution Modes

| 모드 | 특징 | 적합한 상황 |
|------|------|------------|
| **에이전트 팀 모드** | SpawnManager + TaskCreate 기반 | 독립적 작업이 많을 때, 병렬 처리 필요 시 |
| **시니어 에이전트 모드** | 메인 Agent 중심 조율 | 긴밀한 협업 필요 시, 컨텍스트 공유 중요 시 |

### 간소화 가이드

모든 Phase를 항상 수행할 필요는 없습니다:

| 작업 규모 | 권장 Phase |
|-----------|-----------|
| 문서/설정만 추가 | Phase 1, 6 |
| 단일 에이전트 추가 | Phase 1, 3, 6 |
| 에이전트 팀 추가 | Phase 1-6 전체 |
| 워크플로우 변경 | Phase 1, 5, 6 |
